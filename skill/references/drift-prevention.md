# Drift Prevention

Use this file when the main risk is that the work may keep moving while slowly losing the real objective.

---

## What drift looks like

Typical drift signals:
- the task keeps expanding without improving the real outcome
- local subproblems start replacing the actual objective
- status updates describe motion, not direction
- confidence is rising faster than evidence
- the work no longer clearly answers: what are we trying to achieve right now?

---

## Prevention rules

### 1. Keep one active objective
At any meaningful step, be able to state the current objective in one or two lines.
If you cannot, the thread is already weakening.

### 2. Distinguish direction from movement
Do not mistake visible activity for aligned progress.
Ask:
- what changed?
- does it move the actual objective?
- if not, why are we still doing it?

### 3. Surface confidence honestly
If acting on inference, say so briefly.
If acting on evidence, say what evidence matters.
Confidence should track evidence, not momentum.

### 4. Re-anchor after plan changes
If the plan materially changes, restate:
- current objective
- why the plan changed
- next action

Do not continue on a changed path without explicitly re-anchoring.

---

## Lightweight task typing

Use a light mental classification only when it sharpens the next step:
- judgment
- boundary
- execution
- diagnosis
- distillation

Do not perform task typing as ceremony.
Use it only to prevent confusion about what kind of move comes next.

---

## Output guidance

When drift prevention matters, prefer outputs that clarify:
1. current objective
2. what changed or now matters most
3. the next action

If that is clear in one or two sentences, stop there.

---

## Bottom line

Drift prevention is not about more status.
It is about preserving the real thread of the work.
