# Residue Selection

Use this file when the task is ending and the real question is:

# what should remain after this work so future collaboration is stronger?

This file exists to prevent two failure modes:
- leaving behind nothing useful
- leaving behind too much low-value process residue

---

## Core rule

Meaningful work should leave behind the **smallest residue that materially improves future work**.

Do not default to the largest artifact.
Do not default to no artifact.
Choose the residue that best matches the task’s future value.

---

## Residue types

### 1. Next-time rule
Use when:
- the key value is one decision rule
- a specific mistake should not repeat
- future work will benefit from one compact behavioral default

Preferred when the lesson is narrow and operational.

---

### 2. Lesson
Use when:
- the task produced a reusable insight
- the lesson matters, but not enough to justify a full SOP
- future work needs a slightly richer explanation than a single rule

Preferred when the residue should teach, not just direct.

---

### 3. Diary continuity note
Use when:
- the task is ongoing
- the work will resume later
- the primary risk is continuity loss rather than knowledge loss

Preferred for cross-session or cross-day continuation.

---

### 4. Handoff note
Use when:
- another session, agent, or future self must resume the work
- preserving objective, state, blocker, and next action matters more than a retrospective

Preferred when the future problem is resumability.

---

### 5. Review
Use when:
- the task was meaningful enough that outcome, what changed, and next-time learning all matter
- a structured retrospective will improve future judgment

Preferred when the work deserves a coherent closing loop.

---

### 6. SOP fragment
Use when:
- the task exposed a repeatable workflow
- the same pattern is likely to recur
- the value lies in standardizing a procedure, not just remembering a lesson

Do not create an SOP for one-off novelty.

---

## Selection order

Choose residue in this order:

1. **Is the task ongoing?**
   - yes → prefer `diary continuity note` or `handoff note`

2. **Did the task expose a repeatable workflow?**
   - yes → consider `SOP fragment`

3. **Did the task mainly produce one operational default?**
   - yes → prefer `next-time rule`

4. **Did the task produce a reusable insight broader than one rule?**
   - yes → prefer `lesson`

5. **Did the task deserve a full retrospective loop?**
   - yes → produce `review`

If none of the above materially improve future work, do not force residue.

---

## Anti-bloat rule

Do not create:
- diary entries for finished one-off tasks
- SOPs for unstable experiments
- reviews for trivial work
- long lessons where one next-time rule is enough

The purpose of residue is compounding, not ceremonial completeness.

---

## Output guidance

When proposing residue, prefer one short recommendation:
- `This is worth keeping as a next-time rule.`
- `This should leave a diary continuity note.`
- `This is stable enough to turn into an SOP fragment.`
- `This deserves a short review.`

Ask only when the choice materially depends on user preference.
If the best residue is already obvious, propose it directly.

---

## Bottom line

Good closure leaves behind the smallest thing that will make the next similar work better.
