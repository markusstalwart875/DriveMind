# Closure & Compounding

Use this file when the task should leave behind reusable strength after it ends.

---

## The real risk

A task can be completed and still be wasteful if it leaves behind no residue that improves future work.

Typical closure failure:
- no lesson
- no next-time rule
- no reusable handoff state
- no memory worth keeping
- no difference in how the next similar task will be handled

---

## Core rule

Meaningful work should leave behind stronger future defaults.

---

## What counts as compounding residue

Depending on the task, keep the smallest residue that will actually matter later:
- structured review
- reusable lesson
- next-time rule
- memory entry
- handoff note
- SOP fragment when repetition justifies it

Do not create residue just to feel thorough.
Keep only what will improve future judgment or execution.

Read `residue-selection.md` when the real question is *which* residue form should remain.

---

## Review integrity

A useful review should preserve:
- outcome
- what happened
- what changed the result
- whether a boundary or escalation mattered
- reusable lesson
- next-time rule

Use `templates/review-template.md` when a structured review is needed.

---

## Distillation rule

Persist stable lessons, not raw noise.
If the insight will not change future behavior, it probably does not belong in durable residue.

---

## Output guidance

When closure matters, prefer a short but reusable ending:
- result
- lesson
- next-time rule

If the user does not want the full review, preserve the smallest useful version instead of dropping closure entirely.

---

## Bottom line

Closure matters because work should not only finish.
It should strengthen what happens next.
